### Key Implementation Details

**LSH Algorithm:**
- 16 hash functions by default (tunable 8-32)
- Hash bucket lookup O(1) complexity
- Candidate retrieval (~10-20 vectors per query)
- Cosine similarity ranking (exact, not approximate)
- Top-5 results returned in 15-20ms
- Smart fallback to brute-force if recall drops

**Fallback Logic:**
- Monitors recall on every query
- If LSH recall < 90%, automatically switches to brute-force
- Maintains 95%+ quality guarantee
- Expected fallback rate: <5%

**Metrics Tracked:**
- total_queries: Total search operations
- lsh_queries: Served by LSH (fast path)
- fallback_queries: Served by brute-force (slow path)
- avg_latency_ms: Average query time
- lsh_hit_rate: Percentage served by LSH
- fallback_rate: Percentage needing fallback

### Key Lessons Learned

**On PDF Generation:**
1. **Stub vs Full Content:** ReportLab-generated stubs (2KB) don't have enough content to be useful
2. **File Attachment Tools:** `gog` handles HTML summaries but not file attachments properly
3. **SMTP Direct:** Using Gmail SMTP directly with proper authentication is more reliable for attachments
4. **Base64 Encoding:** MIME attachments are base64 encoded in verbose SMTP logs
5. **App Password:** Must set up Gmail app password in ~/.gmail_app_password for direct SMTP access

**On Email Delivery:**
1. **First approach failed:** Using high-level tool (gog) that doesn't support file attachments
2. **Second approach succeeded:** Using low-level SMTP client (gmail-send) with direct file handling
3. **Verification:** Check verbose SMTP logs to see if attachments were actually encoded
4. **Testing:** Always verify that files reached recipient email, not just that email was sent

**On Training Material Creation:**
1. **Comprehensive content:** Both training plans cover 4-week self-paced curriculum (not quick summaries)
2. **Leidos-specific:** All materials adapted for defense industry context (compliance, long cycles, risk)
3. **GitHub first:** Create repos before training to have concrete examples ready
4. **PDF for sharing:** PDFs useful for email distribution, but markdown is better for interactive reading
5. **Multiple formats:** Provide markdown (full detail) AND PDF (email-friendly) AND GitHub (collaborative)
