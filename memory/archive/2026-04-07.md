## OpenClaw 2026.4.2 Update — Multiple Regressions Discovered

The auto-update on April 5 (~18:54 EDT) to gateway version **2026.4.2** introduced
several silent regressions. Logging here as a consolidated reference because they
keep biting us.

### Regression 1: Telegram channel no longer auto-loads (Apr 5, ~20:00 EDT)
- **Symptom:** Telegram channel completely missing from gateway startup sequence
  after restart. No `[telegram]` log lines at all. Bot config in `openclaw.json`
  was correct and unchanged but channel never initialized.
- **Root cause:** 2026.4.2 changed Telegram from a built-in channel to a plugin
  that requires explicit registration in `plugins.allow` and `plugins.entries`.
- **Fix:** Added `"telegram"` to both `plugins.allow` array and `plugins.entries`
  object in `~/.openclaw/openclaw.json`. Restart gateway.
- **Verification:** `openclaw channels list` should show telegram, and gateway
  startup log should include `[telegram]` lines.

### Regression 2: `claude-cli/claude-sonnet-4-6` model unknown (Apr 6, ~05:47 EDT)
- **Symptom:** Both Telegram and Rocket.Chat returning LLM errors like:
  ```
  "Budget 0 is invalid. This model only works in thinking mode."
  ```
- **Root cause chain:**
  1. The `claude-cli/` provider prefix is no longer recognized in 2026.4.2
     ("Missing auth: claude-cli" in gateway logs).
  2. Gateway falls back to `google/gemini-2.5-pro`.
  3. Gemini 2.5 Pro requires thinking mode, but the request had `budget: 0`
     (thinking off), causing the 400.
- **Fix:** Updated the model config to use a properly-prefixed model
  (`anthropic/claude-sonnet-4-6` direct, not via `claude-cli/`).
- **Lesson:** After any gateway update, audit configured model identifiers for
  unrecognized provider prefixes.

### Regression 3: Rocket.Chat plugin SDK breakage (Apr 6 → Apr 7)
- **Symptom:** RC channel in retry loop for 13+ hours overnight. Gateway error:
  `channel exited: Unable to resolve plugin runtime module`. Also: warning that
  `plugins.allow is empty` for the local extension (red herring — refers to local
  ext registration, not the main array).
- **Root cause:** The custom plugin at
  `~/.openclaw/extensions/openclaw-channel-rocketchat/index.ts` is loaded as
  TypeScript, but 2026.4.2 changed how/when TS plugin loading works. Additionally,
  the plugin uses `RuntimeEnv` properties (`channel`, `config`, `logging`) that
  were removed/renamed in the new SDK — same class of issue as the April 2 fix.
- **Status as of 06:34 EDT Apr 7:** Recompiled the plugin to JS (ignoring type
  errors), April 2 patch still in place; restart in progress when this flush
  fired. Needs verification after restart.
- **Action item:** Confirm RC reconnects after restart. If still broken, the
  plugin needs deeper SDK migration to match 2026.4.2.

### Persistent Pattern: Anthropic Sonnet hourly timeouts
- **Symptom:** `anthropic:default` (Sonnet 4.6) timing out at exactly :01 past
  the hour (06:01, 09:01, 14:01, 02:01 EDT) with 61s timeouts. Heartbeat-triggered
  requests are the trigger. Gemini fallback succeeds each time.
- **Impact:** Non-blocking but adds ~60s latency to hourly checks.
- **Possible mitigations (not yet applied):**
  - Bump anthropic timeout to 90s
  - Swap default to `google/gemini-2.5-pro` for heartbeat traffic specifically
  - Investigate whether this is rate-limit related (hourly cluster of requests)

## Checkpoint Recovery Pattern (for future Colab sessions)

When Colab is about to die, the recovery flow is:

1. **Stop training gracefully:**
   ```python
   trainer.args.should_save = True
   trainer.control.should_training_stop = True
   ```
   Or hit the stop button on the running cell — HF Trainer handles `KeyboardInterrupt`
   gracefully.

2. **Force-save:**
   ```python
   trainer.save_model("output/emergency-checkpoint")
   trainer.save_state()
   ```

3. **Push to Drive (survives session death):**
   ```python
   from google.colab import drive
   drive.mount('/content/drive')
   !cp -r output/emergency-checkpoint /content/drive/MyDrive/
   ```

4. **Resume in new session:**
   ```python
   trainer.train(resume_from_checkpoint="path/to/checkpoint")
   ```
   `save_state()` restores LR schedule, RNG, and exact data position.

The H100 notebook bakes all of this in automatically.
